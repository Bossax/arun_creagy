# Markdown Table Edit

## When to use this skill

Use this skill when:

- You need to **add, edit, or delete rows** in an existing Markdown table.
- You need to **normalize or refactor** a Markdown table while preserving data integrity.
- Line-based diff tools (for example `apply_patch`) are **fragile or repeatedly failing** on a table.

This skill is for tables that live in normal Markdown files inside the repo and are **not** generated by another script.

## When NOT to use this skill

Do **not** use this skill when:

- The table is generated from a script or build step (for example a report builder) – in that case, edit the **source data or generator**, not the rendered Markdown.
- You are creating a **tiny table from scratch** and can type it directly in the editor.
- You are converting a **large CSV/Excel** dataset into Markdown (prefer a dedicated converter).
- The requested change affects only non-table content (headings, paragraphs, code blocks).

## Inputs required

Before editing a table, determine:

1. **Target file path**  
   Example: [`ψ/incubate/DCCE/CRDB/CRDB-Evidence-Registry.md`](ψ/incubate/DCCE/CRDB/CRDB-Evidence-Registry.md).
2. **Header anchor text** immediately preceding the table  
   Example: `## Registry table` or `## Trigger table`.
3. **Desired final table** content  
   A complete logical table: header row, alignment row, and all data rows with final values.

## Workflow

### 0. Check for generators first

1. Search the repo for scripts or sources that might generate the table, especially under:
   - `scripts/`
   - `tools/`
   - Project-specific automation folders.
2. If you find a generator that clearly owns the table:
   - **Stop. Do not edit the Markdown table directly.**
   - Update the generator’s source (for example input data, template) and let it regenerate the table.
3. Only continue with this skill if the table is **not** generator-owned.

### 1. Capture the existing table block

1. Use the file-read tool to read the **target file**.
2. Locate the **header anchor** (for example `## Registry table`).
3. From the line after the anchor, capture the contiguous Markdown table block:
   - Starts at the first row beginning with `|`.
   - Ends at the first blank line or the next heading.
4. Treat this span as a **single block** to be replaced.

You do not need to preserve raw spacing or pipe alignment; only the **column order and cell values** matter.

### 2. Build the new table in memory

1. From the current header row, identify the **logical columns** and their order (for example `Evidence ID`, `Artifact`, `Evidence type`, `Date`, ...).
2. Using your inputs, construct a **complete new table** in standard Markdown syntax:

   ```markdown
   | Evidence ID | Artifact | Evidence type | Date | ... |
   |---|---|---|---|---|
   | E-001 | ... |
   | E-017 | ... (updated) ... |
   | E-026 | ... (new) ... |
   ```

3. Ensure every data row has the same number of `|`-separated cells as the header.
4. Keep the table **self-contained**: no extra prose or blank lines inside the block.

### 3. Write the new table to a temporary file

1. Choose a temporary path inside the repo, for example:  
   `.roo/skills/markdown-table-edit/tmp/new-table.md`
2. Use your native file-write tool to write **only the new table block** to that file:

   - The file should contain:
     - Header row
     - Alignment row
     - All data rows
   - No extra headings or surrounding text.

### 4. Replace the entire table block (Full Write)

Do **not** use `apply_patch` or any line-based diff tool to replace the table block. Large hunks of tabular data (many similar `|---|---|` lines) are brittle and prone to context-matching failures.

Instead:

1. **Re-read the entire file** immediately before writing, to avoid stale context (especially if there are dynamic signatures or timestamps elsewhere in the file).
2. In your working copy of the file content (in memory), replace the old table block – from the first header row after the anchor to the last table row – with the newly constructed table block from Step 2.
3. Use your full-file write tool (conceptually `write_to_file`) to **overwrite the entire file** with this updated content.

While writing the whole file is slightly heavier than a localized patch, it is deterministic and bypasses all patch-matching errors for tables.

> Optional: if a deterministic helper script such as [`./.roo/skills/markdown-table-edit/scripts/replace-md-table.ts`](.roo/skills/markdown-table-edit/scripts/replace-md-table.ts) is available and wired into your environment, you may use it instead of manual full-file merging, as long as it performs the same semantics: locate the table under a given anchor and replace that block with the new table.

### 5. Verify after replacement

After running the script:

1. Re-read the target file around the anchor.
2. Confirm that:
   - The new table appears immediately after the anchor heading.
   - The header row and alignment row are correct.
   - All intended rows (for example `E-017`, `E-026`, `T-016`) match the desired final content.
3. If the file includes dynamic signatures or timestamps elsewhere, ensure they still make sense; the script only replaces the table block.

### 6. Fallback when script fails

If the script reports an error (for example anchor not found, or no table after anchor):

1. Re-check the **exact anchor text** and adjust if needed.
2. Inspect the file to confirm that a Markdown table really **starts immediately** after the anchor; if there is intervening prose or comments, you may need to:
   - Move the anchor closer to the table, or
   - Choose a different, more specific anchor heading.
3. Only if the script cannot be adapted, fall back to manual block replacement in the editor:
   - Generate the full new table block as a Markdown snippet.
   - Clearly indicate which lines to replace (for example “from the first `| Evidence ID | ...` line after `## Registry table` through the last table row”).

## Example usage

### Example – Updating CRDB Evidence Registry

Goal: Update the evidence table under `## Registry table` in [`ψ/incubate/DCCE/CRDB/CRDB-Evidence-Registry.md`](ψ/incubate/DCCE/CRDB/CRDB-Evidence-Registry.md) to:

- Mark `E-017` as a historical/retired Phase 1 decision log.
- Add `E-026`, `E-027`, and `E-028` for Thaiwater analysis, concept note, and enhancement plan.

Workflow:

1. Build the complete updated table block in memory, including all existing rows plus updated `E-017` and new `E-026–E-028` rows.
2. Write that table into `.roo/skills/markdown-table-edit/tmp/new-evidence-table.md`.
3. Run:

   ```bash
   bun .roo/skills/markdown-table-edit/scripts/replace-md-table.ts \
     "ψ/incubate/DCCE/CRDB/CRDB-Evidence-Registry.md" \
     "## Registry table" \
     ".roo/skills/markdown-table-edit/tmp/new-evidence-table.md"
   ```

4. Re-open the file and confirm that the evidence table now reflects the updated rows.

Use this same pattern for trigger logs, change logs, or any other Markdown tables that need robust, block-level editing.
