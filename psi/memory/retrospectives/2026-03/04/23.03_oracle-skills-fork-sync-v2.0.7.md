## Key Decisions
- **Preserve local patches first**: treat local installed skills as the “truth you run”, and upstream as the “truth you sync from”.
- **No force-push**: avoid rewriting history; use a sync branch + PR.
- **Handle unrelated histories explicitly**: first-time alignment requires `--allow-unrelated-histories` because the installed skills repo and upstream repo were created independently.

## Honest Feedback (100+ words, 3 friction points)
1) **Repo identity confusion**: `oracle-skills-cli` (installer) vs `~/.roo/skills` (installed skills) are easy to blur together. The workflow only “clicks” once we name them precisely: “upstream source” vs “installed fork snapshot”. Without that clarity, it’s easy to take an action that overwrites the wrong thing.

2) **Unrelated histories is a sharp edge**: the first attempt to merge tags into a repo that began life as a local snapshot predictably fails. It’s correct behavior from Git, but it can feel like a scary blocker. The PR-based sync branch pattern is a clean resolution, but it’s more steps than a typical upgrade.

3) **Windows line endings + interactive tooling friction**: the CRLF warnings add noise, and the merge commit message opening Vim mid-flow is a context switch that can derail people who aren’t comfortable in terminal editors. We handled it, but it’s a reminder that “safe updates” often involve human-factors friction.

## Lessons Learned
- If local environment-specific fixes exist, upstream updates must be treated as **merges** into a **patch-carrying fork**, not as reinstalls.
- For first-time alignment between an installed snapshot repo and an upstream project, plan for `--allow-unrelated-histories` and do it on a dedicated `sync/<tag>` branch.
- A PR-based sync is slower, but it’s the easiest way to review “did we lose our Windows/Git Bash fixes?” before merging.
