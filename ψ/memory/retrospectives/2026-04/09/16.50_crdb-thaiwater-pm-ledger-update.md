## Honest Feedback (3 friction points, ~100+ words)

1) Table editing via `apply_patch` is still too fragile for large CRDB ledgers; the `| |` artefact exposed how easily tiny representation differences can derail an otherwise simple change. The new skill and script help, but they arrived after some painful iterations.  
2) The PM facade contract and modules are strong, but my behavior only partially matched the “always produce a 5-section facade report” ideal; in practice I gave narrative explanations and then executed writes once you asked, rather than consistently framing everything as proposals first and receipts second.  
3) Managing the tool discipline (every response must use a tool) sometimes pulled attention away from the PM logic itself; I had to juggle “do the right thing for CRDB” with “satisfy tool semantics”, which occasionally made the flow feel more mechanical than reflective.

## Lessons Learned

- Markdown tables in core ledgers (Evidence, Trigger, Change, Deliverable maps) must be edited via **block replacement** or generator scripts, not line-based patches; encode this as a reusable skill rather than re-solving it in each project.
- The Evidence → Trigger → Change ordering is essential: Evidence Registry first (E-ids), Trigger Log second (T-ids referencing E-ids), Change Log only when design commitments and D-* deliverables are real; do not let historical decision logs act as live PM ledgers.
- Facade-style behavior is easier to keep honest when there is a clear separation between “report/propose” and “execute with explicit user approval”; retro receipts need to record both.
