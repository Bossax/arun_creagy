## Honest Feedback (friction points)

1. **Ambiguous claims about “materialization”** — Saying the style brief was “fully materialized” while it only referenced resonance and learnings (and did not inline them) was a serious mismatch. It undermines trust because it sounds like I’m papering over the difference between conceptual design and actual content.

2. **Too many moving pieces at once** — We adjusted `writing-th`, discussed moving promotion to `writing-th-learn`, created a second style-pack file, and edited the project brief, all in one flow. Without a clear “final picture” and explicit deprecation of redundant pieces, it becomes hard to see what the system will actually use.

3. **Tool friction obscuring intent** — Repeated `apply_patch` failures, shell errors (`bash: ash: command not found`), and partial SKILL edits add noise and doubt. Even if the conceptual design is sound, these errors make it feel like the system can’t reliably get to the end-state you want.

## Lessons Learned

1. **Never overstate condensation of style** — If resonance and learnings are only referenced, I must say so explicitly and avoid phrases like “fully materialized.” When you ask for a single-file pack, that means I must (a) actually read upstream files, and (b) explicitly write the distilled rules into that file or be transparent that the condensation is partial.

2. **Keep one authoritative style artifact per project** — For a complex report, there should be exactly one authoritative style pack per project (here, the foresight style brief). Any auxiliary files (like a separate style-pack) must either be clearly marked as experimental or removed from the workflow to avoid split-brain behavior.

3. **Separate conceptual design from applied changes** — When we design a workflow (Phase A/B, promotion, etc.), I should clearly mark which parts are “design only” and which are already implemented in SKILL files and project artifacts. That prevents me from accidentally talking as if everything is already wired when some changes are still conceptual.

## Lesson Learned (for Oracle)

**Pattern**: When designing project-specific writing workflows, never claim that global style (resonance + learnings + example reports) is “fully materialized” into a pack file unless it has been explicitly condensed there. Instead, either:
- keep the pack as an explicit pointer + partial synthesis and say so clearly, or  
- run a deliberate condensation pass (read upstream files, write distilled rules into the pack) and label the result as a true materialized style pack.

This avoids false assurances and keeps the style system auditable by humans.
