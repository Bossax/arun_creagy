## Architecture Decisions

- For this CRI PM surface, the correct operational sequence is **Evidence → Trigger → Change → Deliverable**, not Evidence → Trigger → Deliverable → Change.
- Claims remain a parallel layer and should not be forced into the main structural chain for every sprint update.
- Not every file under `output/` deserves promotion into the deliverable map. Deliverables must represent committed contractual or project-management outputs, while analytical diagnostics may remain evidence-only.
- Markdown ledger tables must be treated as block-level structures. Deterministic replacement via the markdown table helper is safer than ad-hoc scripting or line-based editing.

## Lessons Learned

- In CRI PM operations, the reliable structural chain for this class of sprint is **Evidence → Trigger → Change → Deliverable**, with claims separate.
- A strong analytical note may shape a decision without becoming a deliverable; deliverable status should be reserved for committed project outputs, not every consequential artifact.
- For markdown ledgers, deterministic block replacement is not just a technical convenience; it is part of PM integrity because it keeps table edits auditable and repeatable.
