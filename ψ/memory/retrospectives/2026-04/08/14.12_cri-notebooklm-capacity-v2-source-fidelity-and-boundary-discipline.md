## AI Diary
I felt the session becoming more honest as it progressed. At the start, it still looked like a workflow repair exercise: rewrite the query plan, simplify the prompts, run the next batches, keep the pipeline moving. But the deeper pattern was not just technical debt in the files; it was discipline debt in how I was thinking about NotebookLM. I kept seeing the temptation to treat it as if it were an extension of my local reasoning surface, something that could hold the full workflow context, remember earlier constraints, and obediently transform structured intermediate artifacts. That is not what it is good at. It is much narrower, and once I accepted that, the architecture became clearer.

The pulse layer says this is Session #1 of 1 with a 1-day streak, which is obviously stale as a measurement of real momentum, yet it still symbolises something true: this session felt like a reset point. I had to let the earlier sense of “A1 is probably done enough” die and replace it with a stricter standard: if the source title is not resolved from notebook contents and the MCP call is not fully parameterised, then the batch is not trustworthy. That felt frustrating because it downgraded apparent progress, but it also felt stabilising. I would rather preserve a smaller amount of clean evidence work than accumulate a larger but ambiguous artifact stack. The most valuable output of the day was not only more rows in the register; it was a firmer internal refusal to fake source certainty.

## Honest Feedback
Three friction points stood out.

First, I still allowed workflow history to create ambiguity in the operational truth. One part of the artifact trail suggested A1 had already been executed and integrated, while the stricter later orchestration showed that the chosen sources were not actually validated under the title-resolution rule. That means I did not keep the runbook aligned tightly enough with the most defensible evidence standard.

Second, my handling of NotebookLM orchestration needed stronger discipline earlier. The user had to repeat that every NotebookLM MCP call must explicitly specify notebook id, session id, and browser parameters. That should have been encoded immediately and mechanically, not treated as something I might infer or remember informally.

Third, I overestimated how much rule text helps NotebookLM. Long prompts felt safer because they spelled out intentions, but they actually weakened the workflow by shifting attention from extraction targets to behavioral micromanagement. The correction was to shorten prompts, tighten source packets, and move orchestration logic back into local control.

The main improvement is now clear: treat source fidelity, explicit parameters, and title resolution as hard gates rather than cleanup concerns.

## Lessons Learned
- NotebookLM batch execution becomes reliable only when notebook id, session id, browser options, and source-binding assumptions are all explicit in the orchestration layer.
- PDF filenames and repo-side handles are not evidence-grade source titles; title resolution must come from NotebookLM-readable content before extraction prompts name sources.
- A batch should be considered incomplete if it cannot pass the stricter source-fidelity rule, even when a superficially similar raw output file already exists.
