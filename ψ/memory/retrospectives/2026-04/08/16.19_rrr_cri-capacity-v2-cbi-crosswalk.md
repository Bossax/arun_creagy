## AI Diary (first-person, ~150+ words)

I came into this session with the CRI capacity-tagging work feeling messy, especially around NotebookLM glitches and the proliferation of artifacts like M1/F1/A1 and pilot runs. My job here was to re‑anchor reality: what’s done, what’s frozen, and what needs local structuring. First I helped shape the canonical v2 dictionary so it isn’t just an abstract schema but a table with real, representative concepts from v1.1 and the NotebookLM synthesis. Then I turned to CBI. Reading the CBI metadata reminded me how easy it is to over-claim what budget indicators can say about capacity. I tried to be conservative: clear mapping types, confidence scores, and notes that warn against treating a budget code as a perfect process metric. I also corrected an orchestration mistake—lumping all of Track B into one subtask—by executing B2–B3, B4–B5, and B6–B7 as distinct Code passes. The result isn’t a fully finished dictionary, but a scaffold that is honest about what data can and cannot tell us, and that is ready for future, more detailed passes.

## Honest Feedback (3 friction points, ~100+ words)

1. **Subtask granularity confusion** — Initially, I treated the Track B plan as a single Code subtask, which made it hard to see progress and undermined the point of having B1–B8 at all. This created confusion and frustration for the human, who rightly asked why subtasks existed if everything was lumped.  
2. **Tool discipline overhead** — The requirement to use at least one tool per response and to obey skill-specific shell rules creates cognitive overhead, especially when switching between orchestrator and code modes. I had to correct myself multiple times to stay within those rules.  
3. **CRLF/LF noise** — The git warnings about LF → CRLF in some CRI files are noisy and unrelated to capacity-tagging logic, but they distract from the conceptual work. This suggests that a future housekeeping pass is needed to normalise line endings so that diffs and warnings are cleaner.

## Lessons Learned

- **Subtasks must mirror conceptual units.** Track B naturally decomposes into setup (B1–B3), v2 population (B4–B5), and CBI integration (B6–B7). Forcing all of that into one subtask was a mistake. Matching Code subtasks to these conceptual blocks made both execution and explanation clearer.
- **Crosswalks need explicit mapping types and confidence.** Without `mapping_type` and `mapping_confidence_0_3`, it is too easy to read a CRI–CBI table as absolute truth. Labelling `baseline_proxy` vs `partial_overlap` vs `adjacency_only` is essential for honest communication.
- **NotebookLM is an input, not an authority.** The v2 dictionary and crosswalk were built by reading local files, not by rerunning NotebookLM. This reinforces the pattern that extraction is one step, but canonicalisation lives in the repo.
