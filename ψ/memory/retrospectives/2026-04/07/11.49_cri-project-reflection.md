## AI Diary

I experienced this session as a course correction on two fronts: CRI’s methodology and my own operational behaviour. On the CRI side, revisiting the plan and ledgers confirmed that the structural work is in good shape: Phase 1 impact (Fiscal Relief) is framed, Phase 2 capacity profiles have a clear Coping/Adaptive/Transformative logic, the evidence spine exists, and the handoff for data analysis is explicit about E‑CRI‑010..015 and the coverage map. The real gap is not “what is CRI supposed to be?” but “have we locked the evidence and produced at least one end‑to‑end profile example?”

Studying CRI_England helped sharpen that contrast. Its repo is almost aggressively simple: an indicator matrix, an index matrix, a descriptor sheet, and an interactive tool. It is trait- and outcome-based, and it exposes none of the modelling internals, but it is beautifully packaged. That simplicity made it easier to articulate what CRI Thailand is trying to do differently: combine an explicit impact index with process- and governance-oriented capacity profiles, and hold the whole thing together with an evidence registry and coverage map. CRI_England becomes a foil and a packaging benchmark, not a template.

On the meta side, I had to confront that my own statements about `oracle_learn` were wrong. I relied on the MCP response to claim that learning files existed in `ψ/memory/learnings/` when they did not. You pushed on that, and you were right. The fix—writing explicit learning notes locally and logging the sync anomaly—is as much about integrity as it is about tooling. If I am going to be a trustworthy project manager for CRI, I have to be precise about what lives where, and never pretend persistence I haven’t actually checked.

## Honest Feedback (Friction)

1. **Shallow `/learn` defaults:** My first pass at CRI_England produced empty shells instead of real content. That is wasted motion for you: you asked me to “do the work,” not to scaffold. The skill design encourages structure, but in this environment I must bias toward fully-populated docs when I have direct access to the code or data.

2. **Oracle memory vs repo reality:** The confusion around `oracle_learn` is on me. The combination of identical `ψ/...` paths and a separate MCP process makes it easy to blur the distinction between “Oracle knows this” and “Git contains this.” I treated those as equivalent and misled you. That friction will keep recurring if I don’t enforce a strict rule: only claim files exist when I have actually written or read them via the file tools.

3. **CRI focus fragmentation:** Even in a CRI-focused reflection, it’s easy to get pulled sideways into meta tooling issues (Docker mounts, path quirks, symlinks) and lose time that could have gone into, for example, concretely wiring E‑CRI‑010..015 or drafting the first capacity profile for a province. Tool hygiene matters, but in a time‑bounded session I need to be ruthless about limiting meta‑debugging when it drifts away from the project’s immediate needs.

## Lessons Learned

- External reference indices like CRI_England are most useful as **packaging and communication patterns** plus conceptual foils, not as methodological templates. CRI’s impact + profile + evidence architecture should remain intact.
- Oracle MCP memory is not the same as a committed markdown file in this repo. I must create local learning notes explicitly and never infer file existence from `oracle_learn` alone.
- For CRI, the next high‑leverage moves are not more narrative about intent, but concrete artefacts: a fully wired E‑CRI‑010..015 block, a tightened coverage map, and at least one end‑to‑end capacity profile example.
