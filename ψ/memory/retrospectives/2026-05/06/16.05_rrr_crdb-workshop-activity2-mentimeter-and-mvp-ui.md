## AI Diary (first-person, 150+ words)
I felt the tension between making the workshop “smart” and making it “usable.” The first version of the Activity 2 polls still carried a design assumption: that participants would meaningfully choose between Level 1 and Level 2. Once you pointed out the obvious failure mode (“everyone will say depends”), it clicked that the poll was asking a question that is *epistemically correct but operationally useless*. I shifted into a more disciplined mode: every poll has to output something we can build from—an actual distribution that tells us what to put on a front page, what trust signals are mandatory, what output format is minimally acceptable, and what boundary we must protect.

I also noticed how easy it is for MVP discussions to drift into generic system talk. Keeping each MVP question narrowly scoped (briefing front page, catalog needs, decision shield boundary, etc.) felt like a practical way to keep the room anchored in “jobs-to-do” rather than aspiration. Running the Streamlit prototype grounded the work: it’s not just words—there’s a concrete UI that can be pointed at, critiqued, and improved.

## Honest Feedback (100+ words, 3 friction points)
1) **Poll design failure mode surfaced late**: the “depends” outcome is predictable; it would have been better to pre-empt this earlier by stress-testing each question against participant incentives (what answer is safest/easiest). The iteration fixed it, but it cost cycles.

2) **Mixed workshop versions create ambiguity**: there are multiple “design” artifacts (morning master design, v2 master design, facilitation guide). Without a single “this is the canonical for Activity 2” pointer, it’s easy to accidentally cross-wire assumptions (e.g., service levels vs reality audit vs website review).

3) **Prototype run vs prototype relevance**: the Streamlit UI ran successfully, but we did not yet decide *exactly* how it will be used in the workshop (demo timing, which screen, what feedback capture). Without that, it risks becoming a shiny object rather than a facilitation tool.

## Lessons Learned
- “Depends” is not an answer—it is a **signal that the question is underspecified** for decision capture. Good workshop polls constrain the answer space to concrete design requirements.
- For Activity 2, the most valuable polls are those that directly translate into **templates**: front-page elements, minimum trust signals, minimum viable output formats, and explicit boundaries for safe use.
