## Architecture Decisions

1) **MCP tools are the canonical memory interface**
   - Do not build “memory retrieval” into runner scripts.
   - Runners are optional and must be formatting-only.

2) **Two-layer memory model for writing**
   - Stable identity constraints live in resonance:
     - `ψ/memory/resonance/writing-style.md`
     - `ψ/memory/resonance/citation-style.md`
   - Evolving patterns + examples live in learnings tagged `writing_th`.

3) **Outline-first is a control boundary**
   - It prevents premature drafting and keeps the human in charge of structure.

## AI Diary (first-person)

I felt the session repeatedly snap between “designing the right system” and “debugging the wrong interface.” The first drift was into Oracle’s HTTP endpoints — it was tempting because it’s concrete and easy to probe, but it wasn’t the architecture you actually want for an Oracle agent. The correction was clarifying: MCP tools are the proper mental model for memory access; HTTP exists for UIs and non-agent integrations.

The second thing I noticed is how easily a skill spec becomes either too declarative (no retrieval handshake, therefore generic writing) or too procedural (hardcoding tool calls and scripts into SKILL.md). The balance we landed on felt healthy: SKILL.md defines a strict handshake and constraints, but retrieval is explicit via MCP tools, and the evolution loop is first-class. The outline-first stop is also a useful psychological boundary: it shifts the system from “I write” to “we shape.”

I’m also aware that we touched global skill files outside the repo, which creates a split-brain: the repo captures the memory, but the behavior lives in the global skill. That’s acceptable, but it’s a place where discipline matters — naming, registries, and being explicit about what’s global vs project-specific.

## Honest Feedback (3 friction points)

1) **Architecture ambiguity (HTTP vs MCP)**
   We wasted time because the plan mixed two access patterns: Oracle HTTP endpoints and MCP tools. It’s a real footgun because both exist, both work, and only one matches “agent memory retrieval.” The fix is not just “don’t use HTTP,” it’s making MCP retrieval an explicit first step in every skill that relies on memory.

2) **Skill location confusion (repo vs global)**
   The skill being global means edits happen outside the repo working tree. That increases the chance of losing context or forgetting what changed. A pattern that helps is always writing a ψ learning entry when global skills change, and referencing the global file path.

3) **Too many moving parts too early**
   We briefly introduced a TS runner, API probing, and citation extraction all at once. The better sequence is: lock the handshake and constraints first; seed resonance notes next (writing-style + citation-style); only then consider optional automation like `--update`.

## Lessons Learned

- Memory retrieval should be **MCP-first** and explicit in the handshake, otherwise the skill outputs “model weights” rather than “Oracle voice.”
- Keep skill specs declarative but **not passive**: include mandatory retrieval + outline stop + learn-back.
- For writing workflows, resonance vs learnings separation prevents the style spec from becoming a junk drawer.
