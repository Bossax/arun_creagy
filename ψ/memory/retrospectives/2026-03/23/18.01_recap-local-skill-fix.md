I hit three main friction points. First, shell ambiguity in this hybrid Windows/Git Bash + Bun environment made the recap issue harder to diagnose than it should have been; a single `ls -t` behaved differently depending on how it was invoked, and I had to validate outputs repeatedly. Second, the repo layout around global versus project skills is still too implicit; the missing directories in the global repo created a huge status diff that obscured the real problem, and it was easy to lose track of which repo I was modifying. I would benefit from a clearer “current repo” indicator and a sharper convention for where custom skills live to reduce context switching and prevent accidental upstream conflicts.

## Lessons Learned
- On Windows/Bun, use `bash -lc` for deterministic `ls -t` ordering when recency matters.
- Keep project-specific skills under `.roo/skill/` to avoid conflicts with upstream skill repos.
- Validate shell behavior early when file-ordering bugs appear in summaries or reports.
