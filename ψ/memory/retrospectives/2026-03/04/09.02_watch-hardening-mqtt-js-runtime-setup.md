## Honest Feedback
1) Terminal output capture is still brittle: several times the tool-runner returned partial/undefined output, which forces a fallback to “write logs to disk and inspect them.” We mitigated this with stable log files, but it’s still a friction point.

2) There’s a boundary mismatch between “repo tools” and “global skills.” Repo-only scripts can be hardened and committed, but the global `/watch` automation still depends on machine-level setup (MQTT broker availability, browser login, etc.). The current approach is correct (make preflight explicit), but it means the final UX remains multi-step.

3) yt-dlp’s EJS warnings are nuanced: clearing the “no JS runtime” warning revealed deeper “remote components” warnings. This is good (more accurate), but it can be confusing—documentation needs to clearly differentiate “hard blocker” vs “optional hardening.”

## Lessons Learned
- Make “preflight + stable logs” the default for anything multi-dependency (MQTT + yt-dlp + captions).
- On Windows + Git Bash, prefer writing command outputs to files for auditability and to survive terminal output issues.
- yt-dlp needs both a JS runtime installed *and* configured (e.g., `--js-runtimes node`) to avoid deprecated extraction paths.
