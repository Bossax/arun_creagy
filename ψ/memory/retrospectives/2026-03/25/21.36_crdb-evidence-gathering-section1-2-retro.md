## Honest Feedback (frictions, 100+ words)

First friction: patch workflows around long markdown tables are brittle. Multiple `apply_patch` attempts failed because of tiny mismatches in long lines, forcing a switch to `sed`-style search/replace. That consumed attention which should have stayed on evidence content. Second friction: the misalignment between how I treated the v1 markdown outline and how you expected the PDFs to function as canonical text exposed a deeper pattern—if I am not extremely explicit about source-of-truth, it is easy for helper artefacts to quietly become “primary”. Third friction: the environment is hybrid (Windows + Git Bash + Dockerised engine), and small details like path syntax, CRLF handling, and sed quoting added mental overhead. None of these are blockers on their own, but together they made it harder to maintain flow while you were pushing for a very strict, justifiably demanding evidence discipline.

## Lessons Learned

- Helper outlines are never canonical; always re-state and enforce where the true source-of-truth lives (here: interim-report PDFs in `inbox_source`).
- Evidence artefacts (tables, notes, matrices) must carry explicit anchors back to PDF sections and appendices, not just to markdown line references.
- For large markdown tables, prefer systematic, pattern-based transformations over fragile, line-exact patches.
- One-subtask-per-TOR-subsection is a good mental frame: it keeps artefacts scoped, inspectable, and easier to QA against the Evidence Gathering Plan.
