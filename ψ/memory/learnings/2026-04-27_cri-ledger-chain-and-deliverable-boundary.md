# Learning — CRI Ledger Chain and Deliverable Boundary

For this class of CRI PM-ledger update, the operative chain is **Evidence → Trigger → Change → Deliverable**, with claims handled as a separate layer rather than forced into the main structural sequence.

The important boundary lesson is that **not every consequential artifact becomes a deliverable**. A note can be recent, influential, analytically strong, and essential to the reasoning chain while still remaining evidence only. In this session, `2026-04-27_coping-period-fit-analysis.md` shaped the understanding of the next refinement round, but it was correctly kept in the evidence registry rather than promoted into the deliverable map. By contrast, `CRI_Capacity_Tagging_Dictionary_v5.1.md` functioned as the committed governance worksheet outcome and therefore qualified for deliverable treatment.

Procedurally, markdown-ledger tables should be edited via deterministic block replacement under the correct heading anchor, using the markdown table helper rather than ad-hoc patching or custom bulk scripts. This protects both table integrity and PM semantics.
