# Learning: Stakeholder-fit as a session design constraint

Stakeholder analysis only changes outcomes when it is encoded into the workshop design itself.

**Pattern**
- Do not treat “stakeholders to invite” as a separate checklist.
- Define **functional stakeholder clusters** (e.g., oversight/steering, coordinating institution, NMHS/climate service lead, economic/planning, data infrastructure, sensitive data holders, local implementers, DRM operations, private sector demand).
- For each workshop session, specify **minimum representation rules** (which clusters must be present and what role they play in the session).
- Treat any missing cluster as a **documented risk** that constrains what can be claimed as “NFCS-ready,” rather than quietly proceeding.

**Why it matters**
- A workshop can be internally coherent but still fail adoption if key functions are absent.
- TOR requirements like “≥15 organizations” should be interpreted as a **coverage threshold** (aim for 15–20 across clusters), not a raw headcount target.
- Devil’s-advocate objections are best resolved by turning mitigations into mechanics (scope gates, template columns, role separation, resourcing checks), not by adding text at the end.
