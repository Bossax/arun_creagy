# Learning: CRDB canonical planning control surface migration

When planning artifacts drift into fragmented naming (for example, engagement-heavy labels masking architecture and governance tracks), execution clarity declines even if content quality is high. The reliable recovery pattern is to perform a **control-surface migration** rather than incremental renaming.

Effective migration sequence:

1. Define one canonical execution spine artifact as the control center.
2. Define one canonical navigation index artifact as the entrypoint.
3. Split domain operational plans only where implementation decisions diverge materially (CDM, NCAIF service, governance/data standards, MVD/L&D).
4. Move directly superseded artifacts to archive paths, preserving traceability.
5. Update all planning ledgers together (Deliverable, Trigger, Change, Evidence, and plan references).
6. Validate by scanning for old active-path references and resolving all residual links.

Key insight: If supersession is not encoded across ledgers and references in the same session, old mental models persist and the team continues navigating by obsolete artifacts.
