# Lifecycle audit pattern: separate requirements validation from prototype evaluation

When mapping project artifacts to the climate-service development lifecycle (Design Study Methodology), treat **workshop “reality audit” mechanics** as *late-Precondition requirements validation*, not as **Core-phase deployment/evaluation**.

## Why this matters

- A workshop can include strong feasibility checks (access/usability/delay), yet still not be equivalent to “users freely explore a deployed prototype and perform realistic tasks,” which is the Core-phase evaluation signal in the design-study framing.
- Conflating these steps causes **narrative inflation** (“we evaluated a prototype”) and increases the risk of **expert drift**.

## Practical rule

- **Precondition evidence**: interview syntheses, requirements characterization, data/user capacity constraints, workflow mapping, feasibility gating (“safe-to-exchange now vs requires clearance/pilot”).
- **Core evidence**: implemented prototype/service surface + test protocol + observed user tasks + results + iteration log.

## Evidence handling

In strict audits, count only:

1) **Direct production artifacts** for a lifecycle step
2) **Downstream proof** (triggers/claims/deliverables/submissions) that the artifact influenced decisions or outputs
