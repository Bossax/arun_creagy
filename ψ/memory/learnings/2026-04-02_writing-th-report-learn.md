## Word choice and phrasing

- เลือกใช้ "3 จังหวัดชายแดนภาคใต้" แทน "สามจังหวัดชายแดนภาคใต้" เพื่อให้สอดคล้องกับภาษาเชิงทางการและสำนวนราชการที่ใช้ตัวเลขอารบิกในชื่อพื้นที่
- ลดการใช้วลีเชิงปรัชญาและ抽象 เช่น "โอกาสชีวิตเปิดออกหรือปิดลงในทางปฏิบัติ" โดยแทนที่ด้วยประโยคที่อธิบายสถานการณ์เชิงปฏิบัติชัดขึ้น เช่น การเชื่อมเงื่อนไขโครงสร้างกับตัวอย่างการตัดสินใจของครัวเรือนหรือเส้นทางการเรียนรู้ของเด็ก
- เพิ่มความชัดของสรรพนามและกลุ่มเป้าหมาย เช่น เขียนให้เห็นว่ากลุ่มหลักคือ "เด็กและเยาวชนกลุ่มเปราะบาง โดยเฉพาะเด็กกำพร้า" แทนการกล่าวถึงเพียง "เด็กและเยาวชน" แบบกว้าง ๆ
- ใช้โครงสร้างประโยคที่กระชับขึ้นในบางส่วน เช่น การเปลี่ยนจากประโยคยาวหลายอนุประโยคในบทนำ ให้เป็นชุดประโยคสั้นต่อเนื่องที่อ่านง่ายแต่ยังรักษาน้ำหนักเชิงวิเคราะห์

## Semantic and structure patterns

- ฟื้นโครงหัวข้อของรายงานให้กลับมาตรงตาม skeleton ที่กำหนดใน example report เช่น การจัดลำดับบทที่ 1–5 และการเติมหัวข้อย่อยภายใต้แต่ละบทให้ครบ (ที่มาและความสำคัญ, วัตถุประสงค์, กรอบการคาดการณ์อนาคต, ขอบเขต ฯลฯ)
- ใช้ผลการวิเคราะห์ปัญหาจากไฟล์ issue analysis เพื่อกำหนด checklist โครงสร้างและเสียง (กลุ่ม A–F) ก่อนเขียนร่างรวม โดยถือว่า Group A (meta-writing), Group B (ภาษาไทยไม่เป็นธรรมชาติ), Group C (bridge logic), Group D (format/skeleton), Group E (scenario architecture), Group F (evidence) เป็นกรอบกำกับโครง
- จัดกลุ่มเนื้อหาวิเคราะห์ใหม่ให้สอดคล้องกับหัวข้อ skeleton โดยย้ายข้อความที่เคยปะปนกัน (เช่น วิธีการกวาดสัญญาณ, การอธิบายแนวโน้ม, การจัดประเภทสัญญาณ) ไปอยู่ในหัวข้อย่อยที่ชัดเจนตามตัวอย่างใน example report
- เปลี่ยนย่อหน้าที่เคยเล่ากระบวนการเขียนหรือ meta-writing ให้กลายเป็นคำอธิบายบทบาทของบทและหัวข้อในฐานะเนื้อหารายงาน เช่น อธิบายว่าบทที่ 2 ทําหน้าที่ "แปลงหลักฐานให้เป็นแผนที่สัญญาณและปัจจัยขับเคลื่อน" แทนการเล่าเรื่องว่าทีมเขียนบทนี้อย่างไร
- ในบทที่ 2 และ 3 ใช้หัวข้อย่อยจาก example report เป็นโครงเดินเรื่อง เช่น "แนวโน้มชีวิตของเด็กและเยาวชนกลุ่มเป้าหมาย", "แนวโน้มที่มีผลกระทบต่อการเปลี่ยนแปลงของชีวิตเด็กและเยาวชน", "ภาพอนาคตฐาน", "แกนฉากทัศน์" และ "ตัวบ่งชี้ฉากทัศน์" แล้วนำเนื้อหาที่กระจัดกระจายจาก v2 มาจัดเรียงใหม่ให้สอดคล้องกับโครงนี้

### สรุป compressed จาก issue analysis (Groups A–F → รูปแบบเขียนรายงาน)

- กลุ่ม A (meta-writing): ภาษาที่อธิบายกระบวนการเขียน/โครงรายงานต้องถูกย้ายออกจากตัวรายงาน ให้เหลือเพียงประโยคสั้น ๆ ที่ช่วยให้ผู้อ่านเข้าใจบทบาทของบท เช่น “บทนี้ทำหน้าที่…” โดยไม่เล่า logic หลังบ้านของผู้เขียน
- กลุ่ม B (ภาษาไทยและ readability): วลีที่ฟังเป็นการแปล เช่น "โอกาสชีวิตเปิดออกหรือปิดลงในทางปฏิบัติ" หรือคำกว้างอย่าง "ระบบ", "digital" ต้องถูกแทนด้วยภาษาไทยธรรมชาติที่มีตัวอย่างสถานการณ์จริงรองรับ (เช่น เส้นทางออกกลางคัน, ต้นทุนเอกสาร, พื้นที่เสี่ยง)
- กลุ่ม C (logic และ causality): ทุก causal chain สำคัญ เช่น access → trust, security → access, vulnerability → conflict ต้องมีประโยค bridge ที่อธิบายกลไกระดับชีวิตจริงอย่างน้อย 1–2 ประโยค ไม่เขียนข้ามจากต้นเหตุไปผลลัพธ์ตรง ๆ
- กลุ่ม D (structure/format): หัวข้อย่อยบางส่วน (เช่น 2.1.1, objectives) ต้องกลับไปใช้รูปแบบทางการตามแบบรายงานฐาน เช่น ทำ objectives เป็นลิสต์ชัดเจน และให้ส่วนวิธีการกวาดสัญญาณใช้กรอบเดียวกับ example report
- กลุ่ม E (scenario architecture): แกนฉากทัศน์และชุดตัวบ่งชี้ต้องสะท้อนโครงสร้างจริง ไม่ใช้คู่แกนที่ mutually exclusive เกินจริง และต้องแยก belonging/identity-security ออกจาก economic use of identity ในชุด indicator
- กลุ่ม F (evidence/citation): ร่างสุดท้ายต้องเขียนบนสมมติฐานว่าทุก claim สำคัญต้องสามารถ map ไปยัง claim–artifact–source; หากยังไม่มีหลักฐาน ให้เขียนเป็น synthesis ที่ระบุข้อจำกัด หรือใช้ placeholder `[HOLD: ต้องเติมแหล่งอ้างอิงจริง]` แทนการเขียนให้ดูเหมือน fact

## Implications for future drafts

- เมื่อเขียนร่างรวมทั้งเล่มในรอบถัดไป ให้เริ่มจาก skeleton ที่ชัดเจนก่อน (ตาม example report) แล้วค่อยนำเนื้อหาย่อยจากร่างย่อยหรือ artifact มาต่อเติม ไม่ควรให้เนื้อหาขยายจน skeleton เบลอแล้วค่อยย้อนกลับมาจัดโครง
- สำหรับคำหลักที่เสี่ยงกลายเป็นวลีเชิงปรัชญา เช่น "โอกาสชีวิต", "ระบบ", "โลกดิจิทัล" ควรเขียนคู่กับตัวอย่างหรือกรณีในชีวิตเด็กในพื้นที่ 3 จังหวัดชายแดนภาคใต้ทุกครั้ง เพื่อให้ผู้อ่านเข้าใจเชิงปฏิบัติทันทีโดยไม่ต้องตีความเอง
- รักษา separation ระหว่างเนื้อหาสาธารณะและ meta-writing: ถ้าจำเป็นต้องอธิบายตรรกะการจัดโครงหรือกระบวนการเขียน ให้เก็บไว้ในแผนการเขียน (writing plan) หรือบันทึกภายใน ไม่ใส่ในเนื้อหารายงานสำหรับผู้อ่านทั่วไป
- เมื่อมีการเขียนร่างใหม่จาก artifact แยกบท (เช่น ชุด draft2 ของบทที่ 1–3) ให้ตรวจสอบโครงหัวข้อของแต่ละบทเทียบกับ example report ก่อนส่ง รวมทั้งเช็คว่าทุกหัวข้อย่อยที่จำเป็นยังอยู่ครบ และไม่มีหัวข้อใหม่ที่ซ้ำบทบาทกันโดยไม่จำเป็น
