# Lifecycle audit pattern: separate requirements validation from prototype evaluat

Lifecycle audit pattern: separate requirements validation from prototype evaluation

When mapping project artifacts to the climate-service development lifecycle (Design Study Methodology), treat workshop “reality audit” mechanics (access/usability/delay checks, feasibility gating) as late-Precondition requirements validation, not Core-phase deployment/evaluation.

Why:
- A workshop can include strong feasibility checks yet still not be equivalent to observing real users using a deployed prototype to perform realistic tasks.
- Conflating these steps inflates narratives (“we evaluated a prototype”) and increases expert drift risk.

Practical rule:
- Precondition evidence: interview syntheses, requirements characterization, data/user capacity constraints, workflow mapping, feasibility gating (safe-to-exchange now vs requires clearance/pilot).
- Core evidence: implemented prototype/service surface + test protocol + observed user tasks + results + iteration log.

Evidence handling (strict audits): count only (1) direct production artifacts per step and (2) downstream proof (trigger/claim/deliverable/submission) that the artifact influenced decisions/outputs.
