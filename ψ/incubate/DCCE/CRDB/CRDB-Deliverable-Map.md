# CRDB Deliverable Map

## Purpose

This ledger tracks **key deliverable units** for the CRDB project and connects them to triggers, decisions, and evidence. It implements the "deliverable unit" pattern from [`plans/2026-03-27-crdb-oracle-project-work-cycle-architecture-plan.md`](plans/2026-03-27-crdb-oracle-project-work-cycle-architecture-plan.md) and supports end-to-end traceability from question to submission.

## Usage and maintenance

- Focus on **high-leverage deliverables** (reports, packs, templates, schemas) rather than every micro-note.
- Add one row when a deliverable becomes operationally important (drafting, review, or submission).
- When a deliverable supersedes a previous one, add a new row and reference the earlier ID in **Notes**; do not delete older rows.
- Keep links consistent with:
  - triggers in [`CRDB-Trigger-Log.md`](ψ/incubate/DCCE/CRDB/CRDB-Trigger-Log.md)
  - evidence in [`CRDB-Evidence-Registry.md`](ψ/incubate/DCCE/CRDB/output/CRDB-Evidence-Registry.md)
  - decisions in [`phase1_decision_log.md`](ψ/incubate/DCCE/CRDB/output/phase1_decision_log.md)
  - the neutral staging timeline via [`CRDB-Project-History-Timeline.md`](ψ/incubate/DCCE/CRDB/output/CRDB-Project-History-Timeline.md)

## Practical reading order

1. Start here to identify **which deliverables exist** and which are planned.
2. For a given deliverable row, follow:
   - **Related triggers** → [`CRDB-Trigger-Log.md`](ψ/incubate/DCCE/CRDB/CRDB-Trigger-Log.md)
   - **Decision dependencies** → [`phase1_decision_log.md`](ψ/incubate/DCCE/CRDB/output/phase1_decision_log.md)
   - **Evidence basis** → [`CRDB-Evidence-Registry.md`](ψ/incubate/DCCE/CRDB/output/CRDB-Evidence-Registry.md)
3. Use [`CRDB-Submission-Log.md`](ψ/incubate/DCCE/CRDB/CRDB-Submission-Log.md) to see when a deliverable was actually sent or frozen.
