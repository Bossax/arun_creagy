## 1) Invite target structure (who we need in the room)

### 1.1 Participant roles (structure-first; fill exact org names later)

Invite targets should be structured into the following role groups (so headcount allocations and breakout design remain coherent):

1) **Host / sponsor / organizers (DCCE)**
   - Sponsor-facing leadership presence (opening / closing)
   - CRDB coordination team (facilitation, registration, timekeeping)

2) **Providers (data originators)**
   - Agencies that own hazard / exposure / vulnerability / impact datasets relevant to the CRDB service surfaces

3) **Processors / integrators (analysis, synthesis, and publication)**
   - Units responsible for processing/deriving indicators, baselines, models, or summaries (including those who publish)

4) **Users (decision / operations / planning)**
   - Practitioners who consume CRDB outputs for planning, response, budgeting, or adaptation project design

5) **Governance / rails stakeholders**
   - Those who can validate baseline endorsement rules, access rails (open/GDX/internal), and operating rhythm

### 1.2 Invite letter “inputs” checklist (what the letter needs)

- Workshop title (working)
- Date and time window (AM/PM split; see Section 6)
- Onsite-only requirement (per E-021)
- Venue (hotel) and location/map
- Registration process (name, position, organization, email/phone)
- Request: **nominate participants** and confirm headcount per organization by deadline
- Contact point(s) (DCCE coordinator)

## 2) Headcount assumptions and allocation model

### 2.1 Total headcount

- Target total: **80 people** (E-021)
- Mode: **Onsite only** (E-021)

### 2.2 Allocation rule (to be applied once agencies are listed)

Use a controlled allocation scheme to keep breakouts workable:

- **DCCE host/organizers**: 8–12
- **Sponsor leadership / observers**: 3–6
- **External participants**: remainder

Breakout feasibility constraint:

- If using **8 groups × 8 people** (64 seats), reserve remaining seats for plenary speakers/observers/organizers.
- If using **6 groups × ~10 people** (60 seats), keep 20 for plenary/organizers/floaters.

### 2.3 Data to capture for each invited organization

For each organization to invite, capture:

- Role group (Provider / Processor / User / Governance)
- Target headcount (requested)
- Named focal point (primary + backup)
- Any constraints (e.g., needs official letter, clearance lead time)

## 6) Session design implications (breakouts and expected outputs)

### 6.1 Non-negotiable facilitation rule

To honor the workshop stream intent:

- Breakouts must force **synthesis**, not dataset listing.

Rationale anchor: “build, don’t repeat” bridge logic (E-025) plus progress-meeting logistics constraint (E-021).

### 6.2 Proposed breakout structure options (choose based on final participant mix)

Option A — **Role-mixed groups** (recommended if provider/processor/user balance can be ensured)

- 6–8 groups, each with a minimum mix: Provider + Processor + User (plus 1 governance participant if available)

Option B — **Role-specific groups** (fallback if participant mix is skewed)

- Separate groups by role type to ensure comfort, then use plenary to reconcile differences.

### 6.3 Breakout prompts (v0) and required outputs

Each group must submit tangible outputs; do not accept “discussion only.”

1) **Baseline endorsement and trust rules (candidate rules)**
   - Output: 3–5 candidate endorsement rules + who signs + review cadence

2) **Access rails and publishing constraints (Open / GDX / internal)**
   - Output: top constraints + “what is safe to publish now” vs “requires governance gate”

3) **Service tier priorities (what must be usable first)**
   - Output: ranked top 3 service surfaces + minimum viable content/data backing for each

4) **Dependencies and coordination risks**
   - Output: dependency list + risk rating + owner to coordinate (names/roles)

5) **Validation questions for late-May pathway**
   - Output: top 3 validation questions to carry into the next workshop / NFCS validation context
