## 0) Provenance and scope guardrails

This package prepares the minimum execution surface to issue invitations and run logistics for the **TOR 5.3.3 Brainstorming Meeting**.

**TOR anchor (literal)**

- TOR 5.3.3 requires: a **brainstorming meeting** to **exchange data, information products, and needs**, at least 1 time, with **minimum 80 participants**.

**Scope guardrail**

- Invite messaging may reference NFCS as **background ecosystem framing**, but the workshop is framed as a **DCCE/CRDB TOR-facing meeting** (not an NFCS output promise).

### 0.1 Two-part day structure (explicit for invite messaging)

This workshop is explicitly framed as a **two-part day**:

- **Morning (main working session; core workshop value)**: structured exchange of **datasets**, **information products**, and **needs/demand**, and structured **service bundle/blueprint** work.
- **Afternoon (lighter segment; DCCE-owned)**: **DCCE briefing** on **NCAIF + data management direction**, followed by **structured feedback capture** (gallery walk / recap board / short forms + light prioritization).

The afternoon is designed to be **low-demand** and to build directly on the morning’s outputs (via a Bridge Board), not to run another deep co-design or stress-test workshop.

## 1) Invite target structure (who we need in the room)

### 1.1 Multi-hat tagging model (replace single-label roles)

Participants should be tagged using a **multi-hat** approach (overlap allowed). This avoids misclassification where an organization is both provider and integrator, or a user is also a co-producer.

**Suggested tags (use multiple per person as needed)**

- **[Provider]** dataset originator / owner
- **[Integrator/Processor]** derives indicators, models, composites, products
- **[Operator/Implementer]** runs operations or programmes using the outputs (e.g., emergency response, municipal implementation)
- **[User/Decision]** uses outputs for planning/budgeting/policy decisions
- **[Governance/Rails]** access rails, standards, interoperability, data-sharing pathways
- **[Steward/Quality]** data stewardship, metadata, QA/QC

**How to use tags operationally**

- Require each breakout room to include a **mix of tags**, not a mix of “organizations only”.
- Scarce tags (often [Governance/Rails], [Steward/Quality]) should be distributed across rooms (see Section 2.3).

**Representation requirement (invite wording; avoid token participation)**

- Request **knowledgeable and authoritative representatives** who:
  - can speak substantively about their organization’s datasets/products/needs and constraints;
  - can provide actionable views (not only general opinions);
  - can coordinate follow-up (or nominate the right follow-up contact) after the meeting.
- Where possible, encourage **complementary roles** (e.g., technical + stewardship + user/product) rather than one ceremonial attendee.

### 1.2 Minimum participant mix (value-chain coverage)

Invite targets should cover the climate risk and adaptation data value chain:

- dataset providers and stewards
- integrators/processors
- operators/implementers
- end users/decision makers
- rails/governance actors who can explain constraints and pathways

## 2) Headcount assumptions and allocation model

### 2.1 TOR minimum and operational target

- TOR minimum: **≥80 participants**
- Operational target: **80–100 participants** (buffer for no-shows)

### 2.2 Breakout feasibility constraint (default)

- Default breakout model: **max 6 rooms**, **≤10 participants per room**.

Implication: for 80+ participants, not everyone will be in a “deep breakout seat” at the same time. Use:

- rotating participation,
- designated rapporteurs,
- plenary capture moments,
- and/or “observer” seating that can be pulled into breakout discussions when needed.

### 2.3 Scarce-role distribution guidance

Some roles are scarce (e.g., [Governance/Rails] or specific national rails bodies). Guidance:

- Prefer requesting **more than one attendee** from scarce-role organizations (e.g., 2–4) so coverage across rooms is possible.
- If only one representative is available:
  - assign them to the room where rails decisions are most critical, and
  - schedule a plenary “rails Q&A capture” so other rooms can still benefit.

## 3) Per-organization headcount guidance (complementary roles, not single attendee)

To make the exchange meaningful, organizations should send **complementary hats** rather than one person trying to cover all topics.

**Default guidance (request)**

- Most organizations: **2 participants** with complementary tags.
- Key providers/integrators (high data/product relevance): **3–4 participants** (e.g., dataset owner + technical integrator + product lead + stewardship/metadata).

**Examples (tag-based, not org-specific)**

- Provider-heavy org: [Provider] + [Steward/Quality]
- Integrator-heavy org: [Integrator/Processor] + [User/Decision]
- User-heavy org: [User/Decision] + [Operator/Implementer]
- Rails org: [Governance/Rails] + [Integrator/Processor]

## 4) What to prepare (include explicitly in invitation)

Participants should prepare the following (lightweight; can be one page):

1) **Datasets you can exchange / describe**
   - name, owner/steward, coverage, resolution, update frequency, access posture, constraints.
2) **Information products you produce or rely on**
   - dashboards/reports/APIs/apps/briefings; intended users; update frequency; limitations.
3) **Top 3 needs/demand statements**
   - a decision/use case, minimum specs (scale/timeliness), and the blocking constraint.
4) **One contact point** who can follow up after the meeting.

---

## 5) Invitation package structure (one master letter + tailored preparation annex)

To keep the invitation operationally scalable (while still being relevant to different stakeholder clusters), use:

1) **One master invitation letter** (same for all recipients)
   - Explains the two-part day structure (morning main working session; afternoon briefing + feedback)
   - Requests knowledgeable/authoritative representatives and follow-up capability
   - Acknowledges prior engagement (interviews and/or related sessions such as WMO NFCS) as continuity and synthesis, not repetition
2) **One stakeholder-tailored preparation annex**
   - Preparation checklist + targeted questions by stakeholder cluster (providers/stewards, integrators, users/operators, governance/rails)
   - Keeps workload light while increasing contribution quality

## 6) What participants will get (clustered messages; avoid many versions)

### 5.1 Data providers + stewards

- A consolidated view of **who needs what**, with minimum specification hints.
- A structured template that can reduce repeated ad-hoc requests by clarifying metadata, access posture, and constraints.

### 5.2 Integrators/processors

- A clearer map of **available baseline inputs** and their maturity/access constraints.
- Prioritized gap themes that can guide product roadmaps and integration priorities.

### 5.3 Operators/implementers + end users (planning/budgeting/response)

- A documented set of **priority needs/use cases** and candidate service bundle directions.
- A clearer understanding of what can be safely accessed now vs what requires staging/pilots.

### 5.4 Governance/rails stakeholders

- Evidence of recurring constraints and where minimum rails/touchpoints would unlock usability.
- A structured record that can inform later standardisation and access pathway discussions.

**NFCS note (background only)**

- Any NFCS references will be positioned as ecosystem context only, not as a workshop output promise.

## 7) Venue and logistics requirements (execution surface)

Minimum requirements derived from TOR modality options and interactive workshop needs:

- Capacity: **≥80 participants**
- Setup: plenary + ability to run up to **6 breakouts** (rooms or zones)
- Registration desk space
- AV: projector/screen, microphones
- Whiteboards/flipcharts + markers + sticky notes
- Wi‑Fi appropriate for participant count (even if capture is offline)
- Catering: coffee breaks + lunch

Materials/capture:

- Printed agenda + name tags
- Baseline capture sheets (Session 2)
- Service blueprint canvas sheets (Session 3)
- Master tracker for outputs per room

---

## 8) Default breakout model (include in internal briefing)

- **Max 6 rooms**, **≤10 participants per room**, mixed-hat composition.
- If using 2 journeys: **3 rooms per journey**.
- If using 1 journey: 6 rooms using different lenses (datasets/products/needs/gaps/touchpoints/consolidation).

## 9) Dependencies and risks (execution)

Immediate dependencies:

- Final modality (hotel vs hybrid/live stream)
- Final invite target list and headcount requests per organization
- Official letter workflow (signatory + lead time)

Execution risks:

- 80+ participants can overwhelm discussion without:
  - strict room facilitation and rapporteurs,
  - pre-labeled capture templates,
  - and clear rules for breakout capacity vs plenary capture.
