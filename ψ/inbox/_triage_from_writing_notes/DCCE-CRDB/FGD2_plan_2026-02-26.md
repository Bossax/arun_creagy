## FGD1 → FGD2 continuity bridge (talking points)

Condensed from Table 2 in [`src/01_Projects/2025-11_DCCE-CRDB/output/FGD 1 result.md`](src/01_Projects/2025-11_DCCE-CRDB/output/FGD%201%20result.md:56) and mapped to Phase 1 design choices:

|---|---|
| Terminology mismatch / unclear definitions | Decide **glossary scope** + **who owns definitions** (D7). |

| Want preview-before-link/download | Decide Phase 1 **dataset preview standard** in the catalog (D8). |
| PR must review before publishing; comms bottlenecks | Decide **draft → review → publish** workflow and who signs off (D9). |
| Want two-way communication (FAQ / chatbot) | Decide Phase 1 minimum (feedback channel + FAQ; chatbot roadmap) (D10). |

## Decision prompts

### D1 — NCAIF design principle for Phase 1
- Adopt **hybrid**: thematic backbone + light persona entry?

### D1b — Persona priority ordering for Phase 1
- Primary: Policy makers/planners, DDPM/operations support, Local leaders/LAO
- Secondary: Researchers, Communities/Citizens

### D2 — Phase 1 flagship products
- Confirm a short list (e.g., provincial risk profiles, hazard/vulnerability map viewer, post-event impact view, MVP data catalog, glossary/boundary guidance).

### D2b — Exportability requirement
- Confirm which products must be exportable/printable (budget packs, briefing decks, one-pagers)

### D3 — Canonical spatial unit(s) for Phase 1 reporting
- Which boundary datasets are authoritative? How do we crosswalk admin vs LAO? What’s the stance on community boundaries (future)?
	#### D3b — Granularity stance
	- For each flagship product, what is the minimum supported level (province / district / sub-district / municipality/LAO / EA)?

### D4 — Event impact / loss & damage reporting stance
- Acknowledge 1-week aggregation lag? Define what can be published in Phase 1 and how to label uncertainty.

### D4b — Data quality / screening stance (minimum)
- Confirm minimum visible quality flags and validation rules (late reporting, wrong categories, update history)

### D5 — Sensitive data access/publishing rules
- Decide what gets published (aggregated vs restricted) and what legal/accountability pathway is required.

### D6 — Publishing gates (minimum metadata + limitations statement)
- Confirm a minimum “publishable dataset/product” checklist (metadata fields + limitations statement + owner pathway)

### D7 — Glossary + definitions authority (FGD1 continuity)
- What is the **minimum glossary scope** for Phase 1?
- Who is the **authority/owner** for definitions (single owner vs federated with an editor)?
- What is the rule when definitions conflict across agencies?

### D8 — “3-depth” information tiers for Phase 1 (FGD1 continuity)
- Confirm the 3 tiers (policy / practitioners / public) and what changes across tiers:
  - language complexity
  - map/indicator explanation
  - caveats and limitations

### D9 — Dataset preview standard (FGD1 continuity)
- Define minimum “preview” for each dataset/product in the catalog:
  - schema/fields + units
  - sample snippet
  - update cadence
  - owner/contact + request pathway
  - limitations statement

### D10 — Publishing workflow + QA/QC checkpoints (FGD1 continuity)

- Adopt a minimum workflow: **draft → review → publish** (or explicit alternative).
- Decide who reviews what (PR/comms vs domain owner vs data steward) and the SLA target.
- Confirm minimum QA/QC checks before publishing (format validation, boundary version check, timeliness label, limitation statement present).

### D11 — Two-way communication minimum (FGD1 continuity)

- Confirm Phase 1 minimum: feedback form + FAQ, with a clear routing/owner.
- Decide if chatbot is Phase 1 or roadmap (default: roadmap unless resourced).
